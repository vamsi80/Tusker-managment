# Plan to Identify and Fix Data Flickering Issues

## Objective

Identify the root causes of data flickering in the UI and implement fixes to ensure smooth, stable data rendering across the application.

## Current Analysis

Based on repository exploration, data flickering primarily occurs due to:

1. **Real-time Updates**: Frequent Pusher events trigger rapid state changes in Zustand store, causing cascading re-renders
2. **Complex State Management**: Surgical updates in task-cache-store.ts and extensive useEffect chains lead to visual inconsistencies
3. **Component Re-rendering**: Unstable dependencies and improper key props cause unnecessary component remounts
4. **Loading States**: Multiple loading flags and transitions create flicker during data updates

## Key Problem Areas

- `src/lib/store/task-cache-store.ts` - Central state with frequent updates
- `src/app/w/[workspaceId]/_components/realtime-notification-listener.tsx` - Real-time event handling
- `src/components/task/kanban/kanban-board.tsx` - Complex useEffect chains and state updates
- `src/components/task/task-table.tsx` - Multiple re-render triggers
- `src/lib/pubsub.ts` - Event broadcasting causing simultaneous updates

## Implementation Steps

### Phase 1: Stabilize Real-Time Updates

1. Add debouncing to `realtime-notification-listener.tsx` to batch rapid updates
2. Implement optimistic updates for real-time changes to reduce perceived flicker
3. Review and optimize Pusher event handling in `pubsub.ts`

### Phase 2: Optimize State Management

1. Use Zustand selectors in components to prevent unnecessary re-renders
2. Stabilize store update functions with useCallback where needed
3. Review entity merging logic in `upsertTasks` for consistency

### Phase 3: Fix Component Re-renders

1. Stabilize useEffect dependencies using useCallback and useMemo
2. Fix dynamic key props (e.g., in workspace-gantt-client.tsx)
3. Add React.memo to components that receive stable props
4. Consolidate multiple useEffect hooks where possible

### Phase 4: Improve Loading and Transition States

1. Implement skeleton loaders with stable heights to prevent layout shifts
2. Debounce filter/search updates to reduce rapid re-renders
3. Separate loading states for different data sources

### Phase 5: Testing and Validation

1. Add performance monitoring for re-render counts
2. Test real-time updates under high-frequency scenarios
3. Validate fixes across all identified components

## Verification

- Run performance profiling during real-time updates
- Monitor re-render counts in development
- Test with multiple concurrent users to simulate high-update scenarios

## Risks

- Over-optimization may reduce real-time responsiveness
- Changes to state management could introduce bugs if not carefully tested
- Need to balance performance improvements with code maintainability
