# Filter Loading Issue - Analysis & Remediation Plan

## CONFIRMED FINDINGS (After Code Analysis)

Based on your confirmation:

- ✅ **All 3 views affected**: List, Kanban, Gantt + Hierarchy (same codebase, shared logic)
- ✅ **Filter applied but partial data loads**: Old data persists, clearing doesn't help
- ✅ **Seeing old data** (not no data)
- ✅ **No console errors**

## Root Cause Identified

The issue is a **multi-layer cache problem** with 3 specific failure points:

1. **Server Cache Key Instability** - `unstable_cache` returns wrong results due to unstable filter hash
2. **Zustand Cache Key Ignores Filters** - `setProjectTasksCache(projectId, data)` only uses projectId as key
3. **Filter Reset Logic Bug** - Uses stale cache when filters are cleared

---

## Executive Summary

Your filters are not loading properly due to a **cache architecture mismatch** between the server-side caching layer (`unstable_cache`) and the client-side Zustand persistent store. When filters change, the cached data from previous filter states is being returned instead of fresh data, and there's no mechanism to invalidate caches when filter state changes.

### Problem 1: Server Cache Key Instability

**Location**: `src/data/task/get-subtasks-batch.ts:27-43`

```typescript
function getFilterHash(filters: Partial<TaskFilters>): string {
  // BUG: JSON.stringify produces different strings for same logical values
```

**Issue**: `JSON.stringify()` on arrays produces different strings even for same content (different object references).

### Problem 3: Filter Reset Logic Bug

**Location**: `task-table.tsx:766-817`


### Problem 4: No Filter-Based Cache Invalidation

When filters change anywhere in the app (List/Kanban/Gantt), there's no:

1. `revalidateTag()` call for server cache
2. `invalidateProjectCache()` call for Zustand store
3. No mechanism to link filter state change → cache clear → fresh fetch

## Recommended Architecture Fix

### Architecture Change Required

1. **Option A: Filter-Aware Zustand Keys** (Simpler, recommended)

   Change cache keys to include filter hash:

   ```typescript
   // Instead of [projectId], use:
   const cacheKey = `${projectId}-${getFilterHash(filters)}`;
   ```

2. **Option B: Bypass Cache on Filter Change** (Cleaner)

   Never cache filtered results:

   ```typescript
   // In get-subtasks-batch.ts
   const hasActiveFilters = Object.keys(filters).some(k => filters[k]);
   if (hasActiveFilters) {
       return await _getSubTasksByParentIdsInternal(...); // Skip cache
   }
   ```

3. **Option C: Full Architecture Overhaul** (Most robust)

   Implement proper cache tags with filter parameters:

   ```typescript
   revalidateTag(`tasks-${projectId}-filters-${filterHash}`);
   ```

   ```typescript
   function getFilterHash(filters: Partial<TaskFilters>): string {
     const normalize = (v: any) => {
       if (v === undefined || v === null) return null;
       if (Array.isArray(v)) return [...v].sort(); // Sort for stable key
       return v;
     };
     return JSON.stringify({
       status: normalize(filters.status),
       assigneeId: normalize(filters.assigneeId),
       tagId: normalize(filters.tagId),
       search: filters.search?.trim().toLowerCase() || null,
       dueAfter: normalizeDate(filters.dueAfter),

     });
   }
   ```

   ```typescript
   // Add at top of file
   const getFilterHashForStore = (filters: Partial<TaskFilters>): string => {
     const normalized = {
       status: Array.isArray(filters.status)
         ? [...filters.status].sort()
         : filters.status,
       assigneeId: filters.assigneeId || null,
       tagId: filters.tagId || null,
       search: filters.search?.trim().toLowerCase() || null,
       startDate: filters.startDate || null,
       endDate: filters.endDate || null,
     };
     return JSON.stringify(normalized);
   };
   ```

   ```typescript
   getProjectTasksCache: (projectId, filters?: Partial<TaskFilters>) => {
       const filterKey = filters ? getFilterHashForStore(filters) : '__none__';
       const cached = state.projectLists[projectId]?.[filterKey];
       if (!cached) return undefined;
       return {
           tasks: cached.ids.map(id => state.entities[id]).filter(Boolean),
           // ... rest
       };
   ```

6. **Add invalidation method**:
   ```typescript
   invalidateProjectCacheWithFilters: (projectId: string) => set(state => {
       const newLists = { ...state.projectLists };
       if (newLists[projectId]) {
           Object.keys(newLists[projectId]).forEach(k => {
               if (k !== '__none__') delete newLists[projectId][k];
           });
       }
       return { projectLists: newLists };
   }),
   ```

## Summary

### Root Causes (Confirmed):

1. **Zustand cache key** (`setProjectTasksCache`) uses only `projectId` - ignores filters entirely
2. **Server cache key** (`getFilterHash`) uses `JSON.stringify` on arrays - produces unstable keys
3. **Filter reset fallback** at `task-table.tsx:777` returns stale cache when filters cleared

### What Happens Now:

```
User applies filter
  ↓
fetchFiltered() called (correct)
  ↓
API returns data (correct)
  ↓
setProjectTasksCache(projectId, data) ← BUG: No filter key!
  ↓
User clears filter
  ↓
getProjectTasksCache(projectId) ← Returns stale data!
  ↓
UI shows WRONG data ❌
```

### What Should Happen:

```
User applies filter
  ↓
fetchFiltered() called
  ↓
API returns data
  ↓
setProjectTasksCache(projectId, data, filters) ← Include filter key
  ↓
User clears filter
  ↓
invalidateProjectCacheWithFilters(projectId) ← Clear filtered caches
  ↓
Fresh fetch triggered
  ↓
UI shows CORRECT data ✅
```
