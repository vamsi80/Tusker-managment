# Procurement Module — Schema Plan (Production-Ready)

> Grounded in the actual codebase:
> - Prisma client: `import prisma from "@/lib/db"` → `PrismaClient` from `@/generated/prisma/client`
> - Errors: `AppError.NotFound()`, `AppError.Forbidden()`, `AppError.Conflict()` from `@/lib/errors/app-error`
> - Permissions: `getWorkspacePermissions(workspaceId, userId)` from `@/data/user/get-user-permissions`
> - Existing enums: `WorkspaceRole { OWNER, ADMIN, MEMBER, VIEWER, MANAGER }` — add `PROCUREMENT`

---

## Design Decisions

**id vs indentId — Two separate fields:**
- `id` → internal `@id @default(uuid())` — used for all FK joins and Prisma queries
- `indentId` → human-readable `IND-2026-05-0001` — shown in UI, generated by DB trigger
- Same pattern for PO: `id` (UUID) + `poNumber` (`PO-2026-05-0001`)

**Vendor scope → Workspace-scoped** (already in existing schema — keep it). Vendors (GST, contact, bank) are legal entities reused across multiple projects in the same workspace. Project-scoping = duplicate vendor records for every project.

**What is broken in the current schema:**
| Issue | Location | Fix |
|---|---|---|
| `IndentDetails.taskId` not `@unique` | `IndentDetails` | New `Indent` model enforces `taskId @unique` |
| No `VendorQuote` model — vendor assigned directly on `IndentItem.vendorId` | `IndentItem` | Full `VendorQuote` model with `@@unique([lineItemId, vendorId])` |
| No PO workflow status | `PurchaseOrder` | `POWorkflowStatus` enum + `status` field |
| No inspection/damage models | Missing entirely | `DeliveryInspection` + `DamageClaim` added |
| `poNumber`/`key` generated in app (race-condition risk) | App code | DB trigger + sequence table approach |
| No `PROCUREMENT` role in `WorkspaceRole` | Enum | Add `PROCUREMENT` value |

## Step 1: DB-Level ID Generation (Raw SQL Migration)

Create this as a raw SQL migration file at:
`prisma/migrations/<timestamp>_procurement_id_triggers/migration.sql`

```sql
-- Sequence counter tables (month-scoped, atomic)
CREATE TABLE indent_id_seq (
  month_key  CHAR(7) PRIMARY KEY,  -- e.g. '2026-05'
  last_seq   INT NOT NULL DEFAULT 0
);

CREATE TABLE po_id_seq (
  month_key  CHAR(7) PRIMARY KEY,
  last_seq   INT NOT NULL DEFAULT 0
);

-- Indent ID trigger function
CREATE OR REPLACE FUNCTION generate_indent_id()
RETURNS TRIGGER AS $$
DECLARE
  v_month CHAR(7);
  v_seq   INT;
BEGIN
  v_month := TO_CHAR(NOW(), 'YYYY-MM');
  INSERT INTO indent_id_seq (month_key, last_seq)
    VALUES (v_month, 1)
    ON CONFLICT (month_key) DO UPDATE
      SET last_seq = indent_id_seq.last_seq + 1
    RETURNING last_seq INTO v_seq;
  NEW."indentId" := 'IND-' || v_month || '-' || LPAD(v_seq::TEXT, 4, '0');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_indent_id
  BEFORE INSERT ON "indent"
  FOR EACH ROW
  WHEN (NEW."indentId" IS NULL)
  EXECUTE FUNCTION generate_indent_id();

-- PO ID trigger function
CREATE OR REPLACE FUNCTION generate_po_id()
RETURNS TRIGGER AS $$
DECLARE
  v_month CHAR(7);
  v_seq   INT;
BEGIN
  v_month := TO_CHAR(NOW(), 'YYYY-MM');
  INSERT INTO po_id_seq (month_key, last_seq)
    VALUES (v_month, 1)
    ON CONFLICT (month_key) DO UPDATE
      SET last_seq = po_id_seq.last_seq + 1
    RETURNING last_seq INTO v_seq;
  NEW."poNumber" := 'PO-' || v_month || '-' || LPAD(v_seq::TEXT, 4, '0');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_po_id
  BEFORE INSERT ON "purchase_order"
  FOR EACH ROW
  WHEN (NEW."poNumber" IS NULL)
  EXECUTE FUNCTION generate_po_id();
```

> **Why DB trigger, not app-generated?** The `ON CONFLICT + RETURNING` is atomic under concurrent inserts. No distributed lock, no race condition across multiple Next.js instances or Vercel functions.

## Step 4: New Model — VendorMaterialCapability // NEW

Builds a capability registry organically — no manual data entry required.

```prisma
model VendorMaterialCapability {
  id           String   @id @default(cuid())
  vendorId     String
  materialName String   // free text, normalized to lowercase + trim in service layer
  unit         String?
  workspaceId  String
  source       String   @default("MANUAL") // "MANUAL" | "AUTO" (AUTO = added via approved quote)
  createdAt    DateTime @default(now())

  vendor    Vendor    @relation(fields: [vendorId], references: [id], onDelete: Cascade)
  workspace Workspace @relation(fields: [workspaceId], references: [id], onDelete: Cascade)

  @@unique([vendorId, materialName])          // idempotent upsert safe
  @@index([workspaceId, materialName])        // for trigram suggestion queries
  @@map("vendor_material_capability")
}
```

> **Business rules enforced in service layer:**
> 1. `materialName` normalized to `value.toLowerCase().trim()` before every write
> 2. On `QuoteService.approveQuote` → auto-upsert `VendorMaterialCapability` with `source: "AUTO"`
> 3. Vendor suggestions use PostgreSQL `pg_trgm` similarity with threshold `> 0.4` (raw Prisma query in `VendorRepository`)
> 4. Workspace-scoped: capabilities never bleed across workspaces
> 5. `@@unique([vendorId, materialName])` means duplicate inserts are safe (upsert, not error)

### SQL to enable pg_trgm (add to migration)
```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX IF NOT EXISTS idx_vmc_materialname_trgm
  ON vendor_material_capability
  USING gin ("materialName" gin_trgm_ops);
```

### Vendor suggestion raw query (in VendorRepository)
```ts
// src/server/services/procurement/vendor.repository.ts
static async findSuggestedVendors(workspaceId: string, materialName: string) {
  const normalized = materialName.toLowerCase().trim();
  return prisma.$queryRaw<{ vendorId: string; similarity: number; materialName: string }[]>`
    SELECT vmc."vendorId", vmc."materialName",
           similarity(vmc."materialName", ${normalized}) AS similarity
    FROM   vendor_material_capability vmc
    JOIN   vendor v ON v.id = vmc."vendorId"
    WHERE  vmc."workspaceId" = ${workspaceId}
      AND  v."isActive" = true
      AND  v."status" != 'BLACKLISTED'
      AND  similarity(vmc."materialName", ${normalized}) > 0.4
    ORDER BY similarity DESC
    LIMIT 10
  `;
}
```
